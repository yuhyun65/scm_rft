## 하루 세션의 권장 프롬프트 흐름

```
세션 시작
    │
    ▼
[템플릿 A] 세션 시작 컨텍스트 고정
    │  저장소/브랜치/환경값/전 세션 상태 선언
    │
    ▼
작업 유형 판단
    │
    ├─ 단일 이슈 구현 ──────────────→ [템플릿 B] 이슈 원샷 실행
    │
    ├─ 독립 이슈 2~3개 ─────────────→ [템플릿 C] 병렬 이슈 실행
    │
    ├─ 게이트 검증만 필요 ──────────→ [템플릿 D] 7게이트 안정화 실행
    │
    └─ 이슈 구현 + 게이트 ──────────→ [템플릿 B] 내에 게이트 포함됨 (별도 불필요)
    │
    ▼
(반복 — 다음 이슈)
    │
    ▼
[템플릿 E] 세션 종료
    │  종료 조치 실행 + 다음 세션 시작점 요약 자동 출력
    │
    ▼
출력된 "다음 세션 시작점"을 복사해서 다음 날 템플릿 A에 붙여넣기
```

---

## 핵심 설계 원칙

### 1. 원샷 실행 선언
- 구현 + 커밋 + PR + 머지를 1개 프롬프트에 포함
- "자동 처리" 항목을 명시적으로 선언해 단순 확인 Q&A 제거

### 2. 세션 컨텍스트 고정
- 저장소 경로, 브랜치, 환경값을 매 세션 첫 프롬프트에 선언
- 특히 SQL 컨테이너명·env파일·Gateway 정책 경로는 반드시 포함 (게이트 실패의 주요 원인)

### 3. DoD 수치 기반 선언
- 완료 기준을 "게이트 PASS", "exit code 0", "[FAIL] 0건" 등 측정 가능한 수치로 표현
- 모호한 표현("잘 돼야 함")은 사용하지 않음

### 4. 조건부 복구 규칙 내장
- 게이트 실패 유형별 자동 복구 순서를 프롬프트에 포함 (템플릿 D 참조)
- 복구 2회 실패 시 중단+보고 규칙으로 무한 루프 방지

### 5. 의존 관계 명시
- 병렬 실행 가능 여부를 프롬프트에 명시 (템플릿 C)
- 다음 이슈 착수 조건(선행 PR 머지)을 함께 선언

## 비효율 패턴 vs 최적화 패턴 요약

| 비효율 패턴 | 원인 | 최적화 방법 |
|---|---|---|
| "예" 승인 Q&A 반복 | 자동 처리 미선언 | 템플릿 B/C의 【자동 처리】 섹션 |
| 환경 확인 Q&A 반복 | 세션 컨텍스트 미선언 | 템플릿 A 매 세션 사용 |
| 게이트 실패 복구 루프 | 환경값·복구 규칙 미명시 | 템플릿 D의 【고정 환경값】+【복구 규칙】 |
| 저장소 경로 혼선 재작업 | 경로 비선언 | 템플릿 A의 저장소 경로 고정 |
| 커밋/푸시 별도 승인 | 단계 분리 요청 | 자동 처리 선언으로 제거 |
| 다음 세션 준비 Q&A | 종료 요약 없음 | 템플릿 E의 다음 세션 시작점 자동 출력 |
